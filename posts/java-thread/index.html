“프로세스와 쓰레드의 차이를 아세요?”

흔히 듣는 질문이다. 중요하니깐 많이들 물어보는 것이겠지만, 가끔 갑자기 물어보면 제대로 대답하기가 어려운 질문이기도 하다.

프로세스는 실행된 프로그램의 인스턴스이다. 그리고 쓰레드는 프로세스 내의 실행 단위이다. 프로세스 내부에 쓰레드가 존재한다.

둘의 차이는 쓰레드의 특성을 생각해보면 쉽다. 쓰레드는 프로세스 내에서 Code, Data, Heap을 쓰레드끼리 공유한다. 그리고 각 쓰레드는 각자의 Stack은 따로 가지고 있는다.

반면, 프로세스는 Code,Data,Heap, Stack 을 다른 프로세스와 공유하지 않는다. 이것이 멀티 프로세스보다 멀티 쓰레드가 효율적인 이유다.

쓰레드는 Stack을 제외한 메모리를 공유하기 때문에, 프로세스 간의 통신보다 쓰레드 간의 통신비용이 더 저렴하다.

간단히 정리하자면, 멀티 프로세스 환경을 만들기 위해서는 멀티 쓰레드 환경보다 같은 동작을 위해 더 많은 자원을 사용해야한다.

이러한 이유로 멀티 쓰레드 환경에서 프로그래밍을 하게 되는 것이다.

Java 🔗

자바는 쓰레드 언어다. 자바 쓰레드는 JVM에 의해 스케쥴되는 실행단위이다. 자바도 멀티 쓰레드 환경에서 동작하게 되는데, 여기에는 주의해야 하는 점이 있다.

멀티 쓰레드 환경은 쓰레드간에 자원을 공유하기 때문에 race condition에 빠지기 쉽다. 당연히 자바에서도 이런 문제가 발생할 수 있다.

그걸 피하기 위해서 자바에서는 Synchronized 키워드를 제공하고 있다.

Synchronized 🔗

Synchronized는 언어 레벨에서 지원하는 동기화 기법인 Monitor이다. 하나의 쓰레드가 특정 객체에 접근하면 다른 쓰레드는 해당 객체에 접근할 수 없게 된다.

그리고 다른 쓰레드가 접근을 하지 못하면 당연히 성능상의 문제가 생길 수 있다. 실제 프로덕션 환경에서는 Synchronized 를 사용하는 것을 피하는 것이 좋다. 유저 트래픽이 제한적인 경우라면 사용을 고려해볼만 하다.

Volatile 🔗

가시성 문제를 해결하기 위한 volatile keyword도 존재한다. volatile 키워드는 변수를 메인 메모리에 저장하라는 명령어이다. 멀티 쓰레드 환경에서는 보통 변수를 캐시에 저장을 하게 된다. 그 경우 두 쓰레드에서 하나의 변수를 보고 있는 것 같으면서도 서로 값을 다르게 저장하고 있을 수도 있다.

이런 경우를 피하기 위해서 메인 메모리에 값을 저장하라는 의미로 volatile 키워드를 사용한다.

하지만 당연히 cache 보다 main memory를 참조하는 것이 느리기 때문에 성능상의 문제가 생길 수 있어 남용하는 것은 피해야한다.
